# HeadHunter token types and revocation

HH.ru has two different token systems. Mixing them up leads to broken cron jobs.

## Application token (`APPL...`)

- Used for **public** endpoints: `GET /vacancies`, `GET /vacancies/{id}`.
- Optional, but increases rate limits.
- Generated **only** in the application admin panel: https://dev.hh.ru/admin
- **Cannot be refreshed via `/token`.** HH.ru's `POST /token` only supports OAuth authorization-code and refresh-token grants; there is **no `client_credentials` endpoint for application tokens**.
- **Cannot be regenerated by a pure API script.** There is no public API to request a new application token. Automated renewal must log into the web admin UI, which requires email + password + OTP and a browser/CDP flow. The token value is also masked in the admin UI DOM (`data-abs-rule="secrets-redact"`), so extracting it programmatically requires a screenshot/OCR or manual copy.
- Common revocation signal from API: `HTTP 403` with body containing `token-revoked` or `Unrecognized authorization`.
- If revoked or expired: go to https://dev.hh.ru/admin → open the Piramiza application → reveal the token value and copy it, then update `HH_APP_TOKEN` in `~/.hermes/.env` or the script.

## OAuth token (`USER...`)

- Used for **personalized** endpoints: applying to vacancies, messaging, saved searches, user info.
- Obtained via the OAuth flow:
  - Authorization URL: `https://hh.ru/oauth/authorize`
  - Token URL: `POST https://api.hh.ru/token`
- `access_token` has a lifetime (`expires_in` seconds).
- **Can be refreshed** with `refresh_token` via `POST https://api.hh.ru/token` using `grant_type=refresh_token`.
- `refresh_token` is single-use and only valid after `access_token` expires.
- The cover-letter/vacancy-monitoring cron does **not** need an OAuth token unless it also submits applications on behalf of the user.

## What to do when the scraper stops with 403

1. Check the error body. Different 403 reasons need different fixes:
   - **`token-revoked` / `Unrecognized authorization`** — the token was revoked. Renew via `https://dev.hh.ru/admin` and copy the new value.
   - **`bad_authorization`** — the token string is being rejected as malformed/invalid. Before assuming revocation, compare the token in `~/.hermes/.env` byte-for-byte with the value shown in the admin panel. Common cause: visually similar characters (`0` vs `O`, `1` vs `l`, `I` vs `l`). A single wrong character produces `bad_authorization` even though the token looks correct.
2. To compare safely: open the admin panel, take a screenshot, then run a small Python diff between the `.env` value and the value from the screenshot. Fix the mismatched characters and re-test `GET https://api.hh.ru/vacancies` before starting a full Playwright renewal flow.

### Quick verification command

```bash
python3 - <<'PY'
env_value = open('/home/hermes/.hermes/.env').read().split('HH_APP_TOKEN=')[1].split('\n')[0]
ui_value  = 'APPL...'  # paste value from dev.hh.ru/admin screenshot here
print('length', len(env_value), len(ui_value))
print('match', env_value == ui_value)
for i, (a, b) in enumerate(zip(env_value, ui_value)):
    if a != b:
        print(f'diff at {i}: env={a!r} ui={b!r}')
PY
```

Real incident on 2026-07-03: the `.env` token contained `I604P9` (digit `0`), while the admin UI showed `I6O4P9` (letter `O`). API returned `bad_authorization`; fixing the single character restored access without regenerating the token.

3. If the token really is dead, renew it: browser/CDP login to `https://dev.hh.ru/admin` with email + password + OTP, reveal the token, update `HH_APP_TOKEN` in `~/.hermes/.env`, re-run the scraper.

## Tip for cron reliability

Make the scraper detect the revocation signal, set a global flag, skip all further API calls in the run, and print a clear Telegram message with the exact manual fix URL. Do not keep retrying with a dead token — it wastes quota and produces empty reports.

## Pitfall: ambiguous characters in copied tokens

HH.ru application tokens are long random strings of uppercase letters and digits. When copying them manually from the admin UI, characters like `O` (letter) and `0` (digit), or `I`/`l`/`1`, are easy to confuse. Always verify the copied token against the source with a strict string comparison. A token that differs by one character will fail with `bad_authorization`, not `token-revoked`.
